# 大规模图片浏览与语义打标签系统架构方案

本文档旨在为拥有 4 万+ 图片（存储于 SMB 网络路径）且已具备文本描述（`overview`）的数据集，提供一套完整的**高性能前端展示**与**自动化智能打标签**的工程化方案。

---

## 一、 前端高性能浏览方案 (Frontend Architecture)

### 1. 核心挑战
* **DOM 爆炸**：一次性渲染 4 万个 `<img>` 标签会导致浏览器内存溢出并直接崩溃。
* **网络 IO 瓶颈**：SMB 网络路径读取大图极其缓慢，无法支撑前端并发请求。
* **布局抖动**：图片加载完成前尺寸不确定，会导致页面频繁重绘（Reflow）。

### 2. 三层解耦模型架构

#### 第一层：后端 API 服务 (Metadata Service) —— 解决“如何检索”
* **技术实现**：使用 **FastAPI (Python)** 构建轻量级 RESTful API。
* **分页策略**：采用 **游标分页 (Cursor-based Pagination)** 或 `LIMIT/OFFSET` 分页。每次请求仅返回 50-100 条元数据（包含 `id`, `file_path`, `overview`）。
* **作用**：确保前端获取数据的速度不受图片总量影响，实现秒级首屏加载。

#### 第二层：图片代理与缓存层 (Image Proxy & Cache) —— 解决“如何读取”
这是性能优化的核心，通过**“空间换时间”**的策略规避网络 IO。
* **核心模式**：**延迟生成 (Lazy Generation) + 本地 SSD 缓存**。
* **工作流程**：
    1. **请求拦截**：前端请求 `GET /api/thumbnail/{image_id}`。
    2. **缓存命中检查**：后端检查本地快速磁盘（如 C 盘或 SSD）的 `cache/` 目录是否存在该 ID 对应的 `.webp` 文件。
    3. **缓存未命中 (Cache Miss) 处理**：
        * 从 **SMB 网络路径** 读取原始大图。
        * 使用 **Pillow** 进行缩放（Resize）并转码为 **WebP 格式**（极高压缩比）。
        * 将生成的缩略图持久化到本地 `cache/` 目录。
    4. **缓存命中 (Cache Hit)**：直接从本地 SSD 读取文件流返回，实现毫秒级响应。
* **作用**：将昂贵的网络读取转化为廉价的本地磁盘读取，极大缓解 SMB 带宽压力。

#### 第三层：前端虚拟化渲染 (Virtual Windowing) —— 解决“如何展示”
* **技术实现**：使用 React/Vue 配合 **TanStack Virtual** 或 **Vue Virtual Scroller**。
* **核心逻辑**：**虚拟滚动 (Virtual Scrolling)**。浏览器内存中始终只维护当前可视区域内的 ~20 个 DOM 节点，随着用户滚动，动态替换节点内容。
* **作用**：无论数据集多大，前端内存占用和渲染压力始终保持恒定。

## 二、 自动化语义打标签方案 (Automated Tagging Pipeline)

### 1. 核心思路
利用 **“向量化 $\rightarrow$ 聚类 $\rightarrow$ LLM 总结”** 的流水线，将非结构化的文本描述转化为结构化的分类标签。

### 2. 实现流程

#### 第一阶段：特征提取 (Text Embedding)
* **任务**：将 `overview` 文本转换为高维向量。
* **技术**：使用 `sentence-transforms` 或 OpenAI API。
* **结果**：将每一行描述映射为多维空间中的一个点，语义相似的图片在空间位置上彼此靠近。

#### 第二阶段：无监督聚类 (Unsupervised Clustering)
* **任务**：自动发现数据中的“簇”（Cluster）。
* **算法**：使用 **HDBSCAN**（基于密度的聚类算法）。
    * *优势*：不需要预先指定类别数量，且能识别出不属于任何类别的“噪声”图片。
* **结果**：图片被自动归类为若干个不同的簇（例如：簇 0 是“风景”，簇 1 是“文档”）。

####第三阶段：语义总结 (LLM Labeling)
* **任务**：将抽象的数字簇转化为人类可读的文字标签。
* **逻辑**：从每个簇中抽取 5-10 个典型样本，通过 **Claude/GPT-4o** 进行总结：“请根据这些描述，用 1-3 个词概括这个簇的主题”。
* **结果**：将生成的语义化标签（如 `[旅游, 海边]`）写回 `res.sqlite` 的 `tags` 字段。

## 三、 总结与复杂度分析

| 维度 | 前端浏览方案 (Browsing) | 自动化打标签方案 (Tagging) |
| :--- | :--- | :--- |
| **核心目标** | 解决大规模 DOM 与网络 IO 瓶颈 | 实现从非结构化文本到结构化标签的转换 |
| **关键技术** | 分页 API + 图片代理缓存 + 虚拟滚动 | Embedding + HDBSCAN + LLM Summarization |
| **性能瓶变** | 磁盘空间 (Cache) 与 CPU (Resize) | 计算时间 (Embedding) 与 API 成本 (LLM) |
| **复杂度预估** | 中等（重点在于构建代理层） | 高（涉及机器学习模型与向量数据库） |

**该方案通过“空间换时间”解决浏览卡顿，通过“语义建模”解决分类难题，是一套完整且具备工业级扩展性的图片资产管理架构。**
